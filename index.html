二、核心学习内容：学到什么、关键结论是什么

1. 大模型接入：不是"接 GPT"这么简单

1.1 9 种接入方式全景

团队按"控制力—成本—上手难度"三维度，将主流接入方式划分为 9 类：

  1. 官方原厂直连：链路最清晰、模型真实性最高，适合正式生产与核心系统。
  2. 云厂商托管（Azure OpenAI、Claude on Vertex AI）：账单/IAM/网络统一走云平台，适合已有云生态的企业。
  3. MaaS 平台（Amazon Bedrock、Gemini Enterprise Agent）：模型超市 + 统一鉴权 + 评测，适合多模型试点。
  4. 聚合/中转平台：接入灵活、支付方便，但 SLA/透明度风险集中。
  5. 企业自建 AI 网关/模型中台：统一认证、路由、审计、限流、成本归集，是规模化必经之路。
  6. 开源模型自部署（vLLM、TGI、Ollama）：数据自主可控，但算力/运维门槛高。
  7. 本地/边缘部署：适合离线、隐私、终端智能场景。
  8. 私有化一体机：政企/金融/能源等高敏感场景首选。
  9. 网页/SaaS 使用 & 工作流/RPA 接入：适合非技术人员体验与业务快速试点。
关键结论 接入方式选择的核心不是"哪个模型最强"，而是"数据边界、成本可控、系统耦合度"三者平衡。生产环境优先原厂/云厂商；规模化必须建 AI 网关；试点阶段可用 MaaS/中转快速验证。

1.3 Token 成本控制经验

  • 缓存命中：将固定内容前置、批量处理时保持上下文一致，可显著降低 Token 消耗。
  • 多轮对话代价：龙虾/Agent 出现后，Token 消耗呈指数级爆发，需建立用量审计。
  • 价格差异：DeepSeek V4-Pro 输入缓存命中低至 0.025 元/百万 tokens，已开启"分钱时代"。

2. AI 工具链：从"会聊天"到"能交付"

2.1 氛围编程 / Vibe Coding

团队已经形成"先出 Demo → 再反推 PRD → 再迭代交付"的新工作流：

  1. Trae IDE：输入自然语言需求，自动生成完整代码，支持实时预览与对话修改。
  2. Google Stitch：一句话/一张图/语音生成高保真 UI，导出 Figma 或 HTML/CSS。
  3. SVG 优先：生成设计稿优先用 SVG（本质是代码），开发可直接复用。
关键结论 AI 工具已经把"写需求 → 画原型 → 写代码"的线性流程压缩成"对话 → 可运行原型 → 迭代"的闭环。产品经理和设计师的工作重心从"动手做"转向"判断与验收"。

2.3 工具使用安全边界

  • VPN 节点：使用日本等低风控节点，避免频繁切换 IP；建议固定付费节点。
  • 账号策略：推荐谷歌邮箱注册成品账号或海外虚拟信用卡，降低封禁风险。
  • 数据安全：严禁将公司机密、客户隐私、未公开方案输入公共 AI 工具；所有 AI 输出必须人工复核。

3. Agent / 数字员工：从"玩具"到"工具"的落地探索

3.1 龙虾（QClaw / OpenClaw）试用结论

能力表现适用场景
文件/命令/API 操作可执行，但稳定性不高，需人工兜底个人助理、重复办公自动化
新闻汇总/文档处理/会议纪要可快速落地，降低上手门槛行政/运营高频任务
原型设计/复杂逻辑能力不足，存在明显幻觉不建议单独使用
记忆能力三层记忆（短期/日记录/高价值提取）个人工作流养成
关键结论 当前龙虾更适合"个人数字助理"场景，用于处理重复性、规则明确的工作；涉及复杂决策、精细逻辑、对外交付的任务仍需人工介入。

3.2 数字员工平台架构（6.26 研讨会）

团队提出"Agent + 工具"作为最小执行单元，数字员工平台需包含：

  • 角色画像 + 提示词工程
  • RAG 知识库 + Memory 记忆
  • Skill 技能目录 + MCP 标准协议
  • 模型网关 + 成本治理 + 可观测性

落地路线图：

阶段周期目标
第一阶段4 个月单员工闭环：注册/套餐/创建/部署/任务执行，输出可演示 Demo
第二阶段6-8 个月商用试点：渠道接入、知识库、审批流、用量统计、后台运营
第三阶段9-11 个月完整增强：模型网关、API 网关、A2A 多智能体协作、成本治理
关键结论 不建议复刻腾讯 WorkBuddy 的"人海战术 + 全场景覆盖"，应聚焦细分赛道，通过"引擎 + 垂直场景"建立壁垒。

3.3 竞品评测关键结论

产品优势劣势/风险
WorkBuddy多角色、提示词工程较深功能同质化，缺乏 PC 端，体验一般
二元 AI数字分身、个人知识库7 天无理由、客单价高，功能未必超出通用模型
腾讯乐享企业知识库、内容协同偏向内容管理，Agent 执行能力弱
关键结论 数字员工赛道的核心门槛不在"有没有大模型"，而在数据沉淀、提示词工程、执行闭环、成本治理四件事。我们要做的不是另一个 Chatbot，而是"能执行任务的数字员工"。

4. SupplyGuard：第三方组件安全巡检 Agent

4.1 项目背景

客户网站引用的第三方插件（如 BShare、问卷星、社交分享按钮）可能因域名过期/被劫持导致跳转赌博、钓鱼网站。传统人工巡检无法覆盖海量页面与动态加载内容。

4.2 核心能力

检测类型说明典型风险
篡改检测SHA256 hash diff 实时对比脚本被替换为恶意跳转代码
断供检测服务商状态追踪 + DNS 变更告警公司倒闭后域名被接管
失活检测SSL 证书过期 / HTTP 异常证书过期导致 HTTPS 加载失败
漏洞检测已知 CVE 安全漏洞使用存在漏洞的组件版本

4.3 设计成果

已完成 6 页高保真 HTML 原型：

  1. 安全总览仪表盘（风险评分 + 4 维指标 + 告警流）
  2. 组件资产清单（全量脚本管理 + 筛选/排序/导出）
  3. 威胁详情分析（代码 Diff + 影响页面 + 4 步修复向导）
  4. 巡检策略配置（扫描频率 / 告警规则 / Webhook / 白名单）
  5. 站点管理（多站点管理）
  6. 巡检报告（导出/统计）
关键结论 SupplyGuard 已经从"需求讨论"进入"可演示原型"阶段，可直接作为安全产品线孵化，或封装成 SaaS 服务对外销售。

四、经验与反思：踩过的坑、验证过的结论

4.1 已经验证有效的做法

  1. "先 Demo 后 PRD"：用 AI 生成可运行原型再反向推导需求文档，比人工写 PRD 更细致、更全面。
  2. SVG 优先交付：设计稿直接生成 SVG/代码，开发无需重新还原，减少沟通损耗。
  3. 定时任务优于对话：在长流程、多步骤自动化任务中，定时任务比对话交互更稳定。
  4. 国产模型 + 国际模型分层调用：低成本任务用国产，复杂推理用国际，成本与效果兼顾。
  5. 内刊多元化：从资讯播报到 IP 故事、硬核工具分享、生活场景叙事，内容形式持续进化，覆盖团队不同兴趣点。

4.2 需要警惕的问题

AI 能力提升的同时，新的风险也在快速出现。以下是我们结合近期实践和外部趋势整理的重点警惕清单：

#问题应对
1AI 幻觉编造数据、伪造文献、虚构案例频发，所有关键结论必须人工复核，建立"AI 初稿 + 人终审"机制。
2数据安全与隐私泄露公司机密、客户隐私、未公开方案严禁输入公共 AI；优先采用私有化/本地化部署，敏感数据不出域。
3Agent/工具稳定性龙虾/Agent 在复杂逻辑、多步骤任务上仍不稳定，关键流程必须设置人工兜底、异常熔断和回滚机制。
4提示词注入与恶意攻击对外暴露的 AI 接口需做输入清洗、权限控制和审计；避免用户通过提示词绕过安全边界或获取越权信息。
5版权与知识产权风险AI 生成代码、文案、图片可能涉及素材侵权，商用前需核查来源，避免直接复制受保护内容。
6深度伪造与信息污染AI 生成内容越来越逼真，事实性内容必须可追溯来源，重要信息需多源交叉验证。
7成本失控Agent/多轮对话会导致 Token 消耗指数级增长，需设置预算上限、用量审计，并按任务复杂度选择模型梯队。
8合规与监管风险数据出境、算法备案、生成内容标识等法规趋严，需提前关注合规要求，保留完整操作日志。
